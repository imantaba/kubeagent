# kubeagent v3.1 — Design: prioritized `scan`

**Status:** approved design (pre-implementation)
**Date:** 2026-06-22

## Goal

Turn `kubeagent scan` from "show the whole cluster" into a **prioritized problem
report**: highest-severity issues first, the noisy categories (restart history,
CronJobs) moved behind opt-in flags. This refines the v3 output so the default
run is a clean signal of what needs attention.

## Decisions (from brainstorming)

- **Default output = problems only, prioritized.** Healthy-quiet workloads,
  restart-only workloads, and CronJobs are hidden unless a flag opts them in.
- **Two priority tiers:** P1 = infrastructure (nodes + kube-system system
  workloads); P2 = workload/pod failures everywhere else.
- **Generic infra signals** (no bespoke CNI detection): a failed CNI surfaces as
  a NotReady node and/or a degraded kube-system DaemonSet shown by name.
- **`--include-cron`** (default off) reveals CronJobs; **`--include-restarts`**
  (default off) reveals restart-only workloads.
- **Restart-only = `Restarts > 0` and otherwise healthy** (any restarts, not a
  threshold).
- **CronJobs are fully gated** behind `--include-cron` (even a failing CronJob);
  failed *standalone* Jobs remain P2 and show by default.
- A **footer hint** reports how many items are hidden and which flag reveals them.

## Invariants preserved

- **Read-only** (List/Get only); **sequential** (no goroutines); stdlib `flag`
  only; exit codes `0`/`1`.
- **`--explain` egress unchanged:** only structured summary fields + cluster node
  names (when degraded) — never pod IPs/node names from `PodRow`, raw specs, env,
  or secrets.
- No new third-party dependency.

## Priority tiers

| Tier | What | Default? |
|---|---|---|
| **P1** | Cluster-health verdict: nodes NotReady/pressure/cordoned + degraded `kube-system` system workloads (incl. the CNI DaemonSet, shown by name). | Always (the first line). |
| **P2** | Flagged non-cron workloads: detector findings (CrashLoopBackOff, ImagePull, OOMKilled, Pending/Unschedulable), degraded replicas (`ready<desired`), failed standalone Jobs. | Always. |
| restart-only | Healthy workloads (`Running`, `ready==desired`, no findings) with `Restarts > 0`. | Only with `--include-restarts`. |
| CronJobs | All CronJobs (and their rolled-up Job pods), regardless of status. | Only with `--include-cron`. |
| healthy-quiet | `ready==desired`, no findings, no restarts, not a CronJob. | Never shown. |

P1 is the cluster verdict (a separate object, already rendered first). The
visible workload list is P2, then (opted-in) restart-only, then (opted-in)
CronJobs.

## Architecture

A new pure step sits between `inventory.Assemble` and `report`/`explain`:

```text
… → inventory.Assemble → inventory.Prioritize(workloads, opts) → report / explain
clusterhealth.Assess (P1 verdict) ─────────────────────────────┘
```

### `inventory.Prioritize`

```go
// Opts controls which lower-priority categories are included.
type Opts struct {
    IncludeRestarts bool
    IncludeCron     bool
}

// Result is the filtered, prioritized workloads plus the counts of what was
// hidden (for the footer hint).
type Result struct {
    Workloads      []Workload // sorted: P2, then restart-only, then cron
    HiddenRestarts int
    HiddenCron     int
}

func Prioritize(workloads []Workload, opts Opts) Result
```

Classification of each workload into a tier:
- **cron** — `Kind == "CronJob"`.
- **problem (P2)** — not cron and `Flagged()` (findings, `ready<desired`, or
  `Status == "Failed"`).
- **restart-only** — not cron, not a problem, and `Restarts > 0`.
- **healthy-quiet** — everything else.

`Prioritize` keeps problems always; keeps restart-only iff `IncludeRestarts`;
keeps cron iff `IncludeCron`; drops healthy-quiet always. It records the dropped
restart-only / cron counts (over the full input) so the footer can say what's
hidden even when the flags are off. Each kept workload is tagged with a
`Priority` tier (2 problem, 3 restart-only, 4 cron), and the result is sorted by
`(Priority, Namespace, Name, Kind)`.

`Workload` gains `Priority int` (`json:"priority"`).

## Output (`internal/report`)

`PrintInventory` takes the `Result` (verdict still passed separately):

- **text:** the P1 cluster verdict block first (unchanged), then the prioritized
  workloads (P2, then any opted-in restart-only / cron) each rendered as today
  (header + pod rows). When the cluster verdict is `Healthy` **and** the filtered
  workload list is empty, print `No issues found. ✅` after the verdict line
  (when the verdict is `Degraded`, the verdict block already shows the P1
  problem, so the all-clear line is not printed). A **footer hint** prints when
  items were hidden:
  `+3 restarted workloads (--include-restarts) · +5 CronJobs (--include-cron)`
  (only the categories that actually have hidden items, and only the ones still
  off).
- **json:** `{"cluster": {…}, "workloads": [ … ]}` — `workloads` is the filtered,
  prioritized set, each carrying `"priority"`. (The hidden-counts footer is a
  text affordance; JSON simply omits hidden workloads.)

## Wiring & flags (`main.go`)

- Add `--include-cron` and `--include-restarts` (stdlib `flag.Bool`, default
  false); build `inventory.Opts` from them.
- `result := inventory.Prioritize(workloads, opts)`; pass `result` to
  `report.PrintInventory` and `result.Workloads` to `ExplainInventory`.
- Everything earlier in `run` (connect → collect → diagnose → Assemble → Nodes →
  Assess → ScopeNote) is unchanged; the fail-fast `--explain` key check stays.

## `--explain`

`ExplainInventory` summarizes the **same filtered set** that the report shows
(plus the cluster verdict when degraded), so the explanation matches the visible
output. It skips the API call when the cluster is healthy and the filtered list
is empty. The old `Notable`/`notableRestartThreshold` logic is removed —
filtering now happens once in `Prioritize`, and `ExplainInventory` takes the
already-filtered workloads.

## Testing (TDD)

- `inventory.Prioritize` — pure table tests: problems always kept; restart-only
  gated by `IncludeRestarts`; cron gated by `IncludeCron`; healthy-quiet always
  dropped; hidden counts correct; `Priority` tags + sort order; a failed cron is
  gated (not P2) while a failed standalone Job is P2.
- `report` — footer hint text (each combination of hidden categories), section
  ordering, the all-clear `No issues found` path, JSON carries `priority` and the
  filtered set.
- `explain` — `ExplainInventory` summarizes the filtered set; skips when empty +
  healthy cluster; egress guard still holds (no `PodRow` leak).
- `main` — `--include-cron` / `--include-restarts` parse; `--explain` fail-fast
  unchanged.

## Docs

- `README.md` — document the two flags and the prioritized/default behavior;
  egress note unchanged.
- `docs/design.md` — a short note that v3.1 made `scan` a prioritized problem
  report with `--include-cron` / `--include-restarts`.

## Out of scope (explicit non-goals)

- Bespoke CNI/per-node attribution (generic node + kube-system signals only).
- Configurable priority thresholds or custom tiers.
- A flag to restore the full v3 "show everything" view (use both
  `--include-restarts` and `--include-cron`; healthy-quiet stays hidden).
- Changing the detectors or the cluster-health verdict logic.
